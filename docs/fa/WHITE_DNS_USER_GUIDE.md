# راهنمای کاربر WhiteDNS

این راهنما برای کاربری نوشته شده که می‌خواهد بداند هر بخش WhiteDNS چه معنایی دارد،
چگونه باید آن را تنظیم کند، و تغییر هر پارامتر چه اثری روی سرعت، پایداری، مصرف
حجم، باتری و امنیت می‌گذارد.

## این راهنما برای چه کسی است؟

اگر برای اولین بار WhiteDNS را تنظیم می‌کنید، پروفایل وارد می‌کنید، Resolver عوض
می‌کنید، بین Proxy و VPN انتخاب می‌کنید، یا می‌خواهید بفهمید چرا اتصال ظاهرا
برقرار است ولی ترافیک کار نمی‌کند، این راهنما برای شماست.

برای استفاده از WhiteDNS لازم نیست جزئیات داخلی StormDNS را بدانید. اتصال را
لایه‌ای ببینید: سرور، Resolverها، مسیر محلی و شبکه.

مستندات مرتبط:

- [راهنمای عیب‌یابی](WHITE_DNS_TROUBLESHOOTING.md)
- [برنامه QA فارسی](WHITE_DNS_QA.md)

## شروع سریع

برای یک راه‌اندازی تمیز، این ترتیب را رعایت کنید:

1. **Connection Profile** بسازید.
   - دامنه یا دامنه‌های سرور را وارد کنید.
   - کلید رمزنگاری را وارد کنید.
   - نامی بگذارید که بعدا بدانید این پروفایل مربوط به کدام سرور است.

2. **Resolver Profile** بسازید.
   - با یک لیست کوچک و معتبر شروع کنید.
   - Resolverهای تکراری، خصوصی، نامعتبر یا خیلی زیاد را حذف کنید.

3. **Mode** را انتخاب کنید.
   - Proxy برای وقتی است که یک برنامه دیگر می‌تواند از پروکسی محلی استفاده کند.
   - VPN برای وقتی است که می‌خواهید ترافیک برنامه‌های اندروید از داخل WhiteDNS
     عبور کند.

4. اتصال را شروع کنید.
   - وضعیت Startup را بخوانید.
   - Resolver state را بررسی کنید.
   - اگر Route check شکست خورد، اتصال را آماده فرض نکنید.

هنگام تست، فقط یک لایه را تغییر دهید. اگر همزمان سرور، Resolver، MTU، Warmup و
برنامه پروکسی را عوض کنید، مشخص نمی‌شود کدام تغییر مشکل را حل یا بدتر کرده است.

## واژه‌نامه کوتاه

| واژه | معنی |
| --- | --- |
| Connection Profile | پروفایل ذخیره‌شده شامل دامنه سرور و تنظیمات رمزنگاری. |
| Resolver Profile | فهرست ذخیره‌شده Resolverها برای حمل بسته‌های تونل DNS. |
| Proxy Mode | ساخت آدرس پروکسی محلی برای استفاده توسط برنامه دیگر. |
| VPN Mode | استفاده از VPN اندروید برای عبور دادن برنامه‌های انتخاب‌شده از WhiteDNS. |
| Route Check | تست اینکه ترافیک کاربر واقعا از مسیر Proxy یا VPN خارج می‌شود یا نه. |
| Warmup | ترافیک اختیاری برای آماده‌سازی یا گرم نگه داشتن تونل. |
| Diagnostics | خروجی Redacted برای عیب‌یابی بدون افشای Secret. |

## پروفایل اتصال

### Domain(s)

این قسمت مسیر عمومی سرور را مشخص می‌کند. اگر چند دامنه وارد می‌کنید، هر دامنه
باید معتبر و مرتبط با همان سرور باشد.

اثر روی کاربر:

- دامنه اشتباه باعث شکست اتصال یا اتصال ظاهری بدون ترافیک می‌شود.
- چند دامنه می‌تواند برای fallback مفید باشد.
- دامنه‌های نامعتبر باعث تولید کانفیگ خراب می‌شوند و باید قبل از اتصال رد شوند.

فرمت‌های مناسب:

```text
example.com
```

یا:

```text
example.com
backup.example.com
```

یا:

```text
example.com, backup.example.com
```

### Encryption Key

کلید رمزنگاری راز مشترک بین کلاینت و سرور است. اگر کلید اشتباه باشد، ممکن است
برنامه تلاش برای اتصال را نشان دهد، اما ترافیک مفید برقرار نشود.

اثر روی کاربر:

- کلید صحیح برای کارکرد واقعی تونل ضروری است.
- کلید نباید در لاگ، Diagnostics، اسکرین‌شات یا پیام عمومی منتشر شود.
- لینک پروفایل و QR پروفایل شامل این راز هستند.

### Encryption Method

این گزینه روش رمزنگاری مورد انتظار سرور را مشخص می‌کند. مقدار آن باید با سرور
هماهنگ باشد.

اثر روی کاربر:

- روش ناسازگار می‌تواند باعث اتصال ناموفق یا ترافیک بی‌فایده شود.
- تغییر آن بدون دانستن تنظیم سرور معمولا راه‌حل مشکل نیست.

## Resolver Profile

Resolverها بسته‌های تونل DNS را حمل می‌کنند. کیفیت Resolverها برای WhiteDNS بسیار
مهم است.

### چرا لیست خیلی بزرگ همیشه بهتر نیست؟

لیست بزرگ ممکن است شامل این موارد باشد:

- IPهای تکراری.
- Resolverهای خاموش یا کند.
- آدرس‌های خصوصی یا Loopback که روی دستگاه کاربر معنی ندارند.
- Resolverهایی که روی شبکه خاصی فیلتر یا کند هستند.

اثر روی کاربر:

- Startup کندتر می‌شود.
- مصرف آپلود بالا می‌رود.
- اتصال ممکن است ظاهرا برقرار شود ولی ترافیک واقعی کار نکند.
- لاگ‌ها شلوغ‌تر و عیب‌یابی سخت‌تر می‌شود.

### پیشنهاد عملی

- با یک لیست کوچک‌تر و تمیز شروع کنید.
- بعد از وارد کردن لیست، تعداد نهایی Resolverهای معتبر را بررسی کنید.
- اگر اتصال کار نکرد، اول Resolverها را عوض کنید، نه همه تنظیمات را با هم.

## حالت Proxy

در حالت Proxy، WhiteDNS یک پروکسی محلی ایجاد می‌کند.

### Listen IP

مشخص می‌کند پروکسی روی چه آدرسی گوش کند.

- `127.0.0.1` یا Loopback: فقط خود دستگاه می‌تواند وصل شود.
- آدرس شبکه محلی: دستگاه‌های دیگر هم ممکن است بتوانند وصل شوند.

اثر روی کاربر:

- Loopback امن‌تر است.
- آدرس قابل دسترسی از LAN نیاز به احراز هویت دارد.
- اگر Listen IP اشتباه باشد، برنامه‌های دیگر نمی‌توانند به پروکسی وصل شوند.

### Listen Port

پورتی است که کلاینت‌های SOCKS به آن وصل می‌شوند.

اثر روی کاربر:

- اگر پورت توسط برنامه دیگری اشغال شده باشد، Proxy شروع نمی‌شود.
- پورت باید دقیقا در برنامه همراه وارد شود.

### HTTP Proxy Bridge

بعضی برنامه‌ها HTTP proxy می‌خواهند، نه SOCKS. HTTP Bridge این فاصله را پر می‌کند.

اثر روی کاربر:

- سازگاری با برنامه‌های بیشتر.
- مصرف منابع بیشتر نسبت به فقط SOCKS.
- باید محدودیت تعداد Client، اندازه Header و Timeout رعایت شود.

### SOCKS5 Authentication

وقتی پروکسی روی شبکه محلی در دسترس است، نام کاربری و رمز عبور غیرخالی ضروری است.

اثر روی کاربر:

- بدون احراز هویت، دیگران در همان شبکه می‌توانند از پروکسی سوءاستفاده کنند.
- رمز را فقط وقتی لازم است کپی کنید و آن را عمومی نکنید.

## حالت VPN

در حالت VPN، WhiteDNS از سرویس VPN اندروید استفاده می‌کند.

### Split Tunnel

مشخص می‌کند کدام برنامه‌ها از تونل عبور کنند.

اثر روی کاربر:

- می‌توانید فقط برنامه‌های لازم را از تونل عبور دهید.
- اگر برنامه‌ای در Split Tunnel انتخاب نشده باشد، ممکن است اتصال WhiteDNS سالم
  باشد ولی آن برنامه از تونل استفاده نکند.

### IPv6 Strategy

رفتار IPv6 را مشخص می‌کند.

- Block: امن‌تر برای جلوگیری از نشت IPv6.
- Bypass with warning: ممکن است سازگاری بهتر باشد، اما IPv6 از تونل عبور نمی‌کند.
- Experimental route: برای تست شبکه‌های خاص، با ریسک ناسازگاری.

### VPN MTU

MTU اندازه بسته‌های شبکه در مسیر VPN است.

اثر روی کاربر:

- `1280`: سازگارتر، مخصوص شبکه‌های سخت‌گیر.
- `1400`: تعادل بین سازگاری و کارایی.
- `1500`: ممکن است سریع‌تر باشد، ولی روی بعضی شبکه‌ها مشکل ایجاد می‌کند.
- Custom: فقط وقتی استفاده کنید که دلیل مشخص دارید.

## Traffic Warmup

Warmup و Keepalive می‌توانند به آماده شدن تونل کمک کنند، اما هزینه دارند.

### Off

هیچ ترافیک عمدی برای گرم نگه داشتن اتصال ارسال نمی‌شود.

مناسب برای:

- کمترین مصرف حجم.
- کمترین مصرف باتری.

### Startup-only

در شروع اتصال چند Probe ارسال می‌شود و بعد متوقف می‌شود.

مناسب برای:

- حالت پیش‌فرض کم‌هزینه.
- کاربرانی که نمی‌خواهند آپلود مداوم ببینند.

### Balanced

در صورت نیاز اتصال را گرم نگه می‌دارد، ولی اگر ترافیک واقعی اخیرا دیده شده باشد
نباید Keepalive اضافه ارسال کند.

مناسب برای:

- شبکه‌هایی که کمی ناپایدار هستند.
- کاربرانی که تعادل بین پایداری و مصرف می‌خواهند.

### Aggressive

ترافیک بیشتری برای حفظ آمادگی ارسال می‌کند.

مناسب برای:

- شبکه‌های سخت و ناپایدار.
- فقط وقتی کاربر مصرف حجم و باتری بیشتر را قبول دارد.

## تنظیمات پیشرفته

### MTU و اندازه بسته‌ها

Min Upload، Max Upload، Min Download و Max Download اندازه بسته‌های تونل را کنترل
می‌کنند.

اثر روی کاربر:

- اندازه کوچک‌تر: سازگاری بیشتر، احتمال سرعت کمتر.
- اندازه بزرگ‌تر: امکان سرعت بهتر، احتمال شکست روی شبکه‌های محدود.

### Duplication

Upload Dup و Download Dup بسته‌ها را تکرار می‌کنند.

اثر روی کاربر:

- تحمل Packet loss بهتر می‌شود.
- مصرف آپلود و فشار روی Resolver بیشتر می‌شود.

### Compression

Upload Compress و Download Compress داده‌ها را فشرده می‌کنند.

اثر روی کاربر:

- ممکن است مصرف را کم کند.
- CPU و باتری بیشتری مصرف می‌کند.
- برای داده‌هایی که از قبل فشرده هستند، سود کمی دارد.

### DNS Fragments

مشخص می‌کند داده‌ها چقدر خرد شوند.

اثر روی کاربر:

- Fragment بیشتر می‌تواند با محدودیت اندازه سازگارتر باشد.
- Fragment زیاد تعداد درخواست‌ها و مصرف آپلود را بالا می‌برد.

### Worker، Queue و Retry

این‌ها رفتار داخلی Runtime را کنترل می‌کنند.

اثر روی کاربر:

- Worker بیشتر می‌تواند روی دستگاه قوی سرعت را بهتر کند.
- Worker و Queue زیاد می‌تواند CPU، RAM و باتری بیشتری مصرف کند.
- Retry زیاد در شبکه خراب، مصرف حجم را بالا می‌برد و تشخیص شکست را دیرتر می‌کند.
- Timeout کوتاه‌تر شکست را سریع‌تر نشان می‌دهد.
- Timeout بلندتر به شبکه کند فرصت بیشتری می‌دهد.

## وضعیت اتصال و Metrics

### Startup Progress

نشان می‌دهد اتصال در کدام مرحله است. اگر برای مدت طولانی ثابت ماند، باید Timeout
یا خطای مشخص نمایش داده شود.

### Resolver State

نشان می‌دهد Resolverها مفید هستند یا نه. اگر Resolverها خراب باشند، سرور سالم هم
ممکن است به نظر خراب برسد.

### Route Check

مهم‌ترین تست برای کاربر است: آیا ترافیک واقعا از مسیر محلی Proxy یا VPN خارج
می‌شود؟

اگر Route Check شکست خورد، برنامه نباید اتصال را کاملا آماده نشان دهد.

### Upload و Download

آپلود می‌تواند شامل درخواست‌های DNS، Warmup، Keepalive و ترافیک واقعی کاربر باشد.
اگر آپلود زیاد است ولی دانلود تقریبا صفر است، معمولا باید این موارد بررسی شوند:

- سلامت سرور.
- کیفیت Resolverها.
- Route Check.
- Warmup mode.

## Diagnostics و Log

Diagnostics برای کمک به عیب‌یابی است، نه برای اشتراک‌گذاری رازها.

خروجی امن باید این موارد را Redact کند:

- کلید سرور.
- رمز SOCKS.
- لینک پروفایل.
- Secrets داخل TOML.
- مسیر فایل‌های Runtime.
- لاگ‌های خام حساس.

## Export و QR

Profile Export باید قابل استفاده بماند. یعنی متن Export و QR باید مقدار واقعی
و Importable داشته باشند. اما چون این مقدار شامل راز اتصال است، باید هشدار واضح
نمایش داده شود.

قاعده ساده:

- Profile Export برای دستگاه یا فرد مورد اعتماد است.
- Diagnostics برای پشتیبانی عمومی است و باید Redacted باشد.

## مسیر عیب‌یابی پیشنهادی

وقتی اتصال کار نمی‌کند، این ترتیب را دنبال کنید:

1. سرور: دامنه، کلید، اعتبار و ظرفیت سرور.
2. Resolverها: لیست کوچک، معتبر، بدون تکرار و بدون IP خصوصی.
3. مسیر محلی: آدرس و پورت Proxy یا تنظیم Split Tunnel در VPN.
4. شبکه: Wi-Fi، Mobile Data، اپراتور، IPv6، MTU و باتری.

همزمان چند تنظیم را تغییر ندهید؛ در غیر این صورت نمی‌فهمید کدام تغییر مشکل را
حل یا بدتر کرده است.
