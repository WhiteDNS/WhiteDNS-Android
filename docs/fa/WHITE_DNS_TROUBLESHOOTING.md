# راهنمای عیب‌یابی WhiteDNS

این راهنما برای زمانی است که WhiteDNS درست کار نمی‌کند یا وضعیت برنامه مبهم است.
هر بخش یک مسیر مرحله‌ای دارد. ترتیب مراحل را حفظ کنید و همزمان چند تنظیم را تغییر
ندهید.

## قبل از شروع

این موارد را آماده کنید:

- نام پروفایل اتصال فعال.
- نام Resolver Profile فعال.
- حالت اجرا: Proxy یا VPN.
- وضعیت آخرین اتصال.
- پیام خطا یا وضعیت Startup.
- Diagnostics به صورت Redacted، نه لینک پروفایل یا کلید خام.

## وصل می‌شود ولی هیچ برنامه‌ای باز نمی‌شود

این مشکل معمولا یعنی یکی از لایه‌ها کار نمی‌کند، حتی اگر بخشی از Runtime فعال شده
باشد.

### مرحله 1: سرور را بررسی کنید

- دامنه سرور را دوباره چک کنید.
- کلید رمزنگاری را چک کنید.
- اگر چند دامنه دارید، با یک دامنه سالم تست کنید.
- اگر سرور محدودیت حجم، اعتبار یا ظرفیت دارد، آن را جداگانه بررسی کنید.

نتیجه مورد انتظار:

- اگر سرور مشکل دارد، قبل از تغییر Resolver یا MTU باید همان حل شود.

### مرحله 2: Resolverها را کوچک و تمیز کنید

- از یک لیست خیلی بزرگ شروع نکنید.
- Resolverهای تکراری را حذف کنید.
- IPهای خصوصی، Loopback یا Link-local را حذف کنید.
- اگر Resolverها پورت دارند، مطمئن شوید برنامه از آن فرمت پشتیبانی می‌کند.

نتیجه مورد انتظار:

- Startup باید سریع‌تر و لاگ‌ها قابل فهم‌تر شوند.

### مرحله 3: مسیر محلی را بررسی کنید

در حالت Proxy:

- آدرس و پورت نمایش داده‌شده در WhiteDNS را دقیقا در برنامه همراه وارد کنید.
- اگر HTTP Bridge استفاده می‌کنید، پورت HTTP را با پورت SOCKS اشتباه نگیرید.
- اگر Listen IP غیر Loopback است، SOCKS Auth باید با نام کاربری و رمز غیرخالی
  فعال باشد.

در حالت VPN:

- مطمئن شوید برنامه هدف در Split Tunnel حذف نشده است.
- مجوز VPN فعال باشد.
- اعلان Foreground Service فعال باشد.

نتیجه مورد انتظار:

- اگر Route Check شکست بخورد، برنامه نباید آماده فرض شود.

### مرحله 4: شبکه را عوض کنید

- یک بار با Wi-Fi تست کنید.
- یک بار با Mobile Data تست کنید.
- اگر امکان دارد اپراتور دیگری را تست کنید.
- IPv6 Strategy و MTU را با مقدارهای محافظه‌کارانه‌تر امتحان کنید.

## درصد اتصال گیر می‌کند یا Startup طولانی می‌شود

### علت‌های رایج

- Resolverهای کند یا خراب.
- سرور در دسترس نیست.
- Timeout خیلی بلند است و شکست دیر نمایش داده می‌شود.
- Warmup یا Retry بیش از حد تهاجمی است.
- شبکه DNS را کند یا محدود کرده است.

### مسیر تست

1. Resolver list را کوچک کنید.
2. Warmup را روی Startup-only بگذارید.
3. MTU را روی 1280 یا مقدار محافظه‌کارانه بگذارید.
4. با یک شبکه دیگر تست کنید.
5. اگر هنوز گیر می‌کند، Diagnostics Redacted بگیرید.

نتیجه مورد انتظار:

- وضعیت نباید برای همیشه گیر کند.
- باید Timeout یا خطای مرحله‌ای نمایش داده شود.

## آپلود زیاد است ولی دانلود کم است

آپلود در تونل DNS لزوما به معنی ارسال فایل یا داده شخصی نیست. ممکن است شامل
درخواست‌های DNS، Warmup، Keepalive، Retry یا Resolver probing باشد.

### مواردی که باید بررسی شود

1. Warmup Mode
   - Off: کمترین ترافیک عمدی.
   - Startup-only: مناسب برای شروع کم‌هزینه.
   - Balanced: تعادل بین پایداری و مصرف.
   - Aggressive: پرمصرف‌تر.

2. Resolverها
   - لیست بزرگ و خراب می‌تواند آپلود را زیاد کند.
   - Retry روی Resolverهای خراب مصرف را بالا می‌برد.

3. سرور
   - اگر سرور جواب مفید ندهد، درخواست‌ها تکرار می‌شوند ولی دانلود مفید کم می‌ماند.

4. Duplication و Fragment
   - Duplication و Fragment زیاد تعداد درخواست‌ها را بالا می‌برد.

### اقدام پیشنهادی

- Warmup را Startup-only کنید.
- Resolver list را کوچک و تمیز کنید.
- Duplication را محافظه‌کارانه تنظیم کنید.
- Aggressive warmup را فقط برای تست کوتاه استفاده کنید.

## برنامه بعد از مدتی در پس‌زمینه قطع می‌شود

### موارد اندروید

- Battery optimization ممکن است سرویس را محدود کند.
- اعلان Foreground Service باید فعال باشد.
- بعضی دستگاه‌ها محدودیت‌های سازنده دارند.
- تغییر Wi-Fi/Mobile Data ممکن است نیاز به Recovery داشته باشد.

### مسیر تست

1. Battery optimization را برای WhiteDNS محدود نکنید.
2. اعلان‌ها را فعال کنید.
3. در حالت VPN، تنظیمات Always-on VPN را بررسی کنید.
4. Wi-Fi را به Mobile Data تغییر دهید و وضعیت Recovery را ببینید.
5. دستگاه را قفل کنید و بعد از چند دقیقه باز کنید.

نتیجه مورد انتظار:

- Recovery باید محدود، Debounced و با همان Session انجام شود.
- Callbackهای قدیمی نباید اتصال متوقف‌شده را دوباره روشن کنند.

## خطاهای Resolver

### نشانه‌ها

- Startup کند.
- لاگ‌های خطای زیاد.
- دانلود کم.
- اتصال ظاهری بدون باز شدن برنامه‌ها.

### راه‌حل

- Resolverهای نامعتبر را حذف کنید.
- Resolverهای خصوصی و Loopback را حذف کنید مگر اینکه دقیقا می‌دانید چرا لازم‌اند.
- لیست را Deduplicate کنید.
- از چند Resolver سالم شروع کنید، نه هزاران مورد نامعلوم.

## مشکل با چند دامنه

فرمت‌های قابل فهم:

```text
example.com
backup.example.com
```

یا:

```text
example.com, backup.example.com
```

اگر چند دامنه کار نمی‌کند:

1. هر دامنه را جداگانه تست کنید.
2. فاصله، کوتیشن و جداکننده‌های اضافی را حذف کنید.
3. مطمئن شوید همه دامنه‌ها برای همان سرور تنظیم شده‌اند.
4. Export و Import پروفایل را تست کنید تا دامنه‌ها حفظ شوند.

## متن‌ها یا Labelها ناپدید می‌شوند

### مسیر تست

1. بین تب‌های Connect، Profiles، Logs و Guide چند بار جابه‌جا شوید.
2. اندازه فونت اندروید را بزرگ کنید.
3. تم یا حالت نمایش دستگاه را تغییر دهید.
4. برنامه را به پس‌زمینه ببرید و برگردانید.

نتیجه مورد انتظار:

- متن‌ها باید قابل مشاهده بمانند.
- دکمه‌ها و هشدارها نباید روی هم بیفتند.
- نام‌های طولانی باید Wrap یا کوتاه شوند.

## چه چیزی را برای پشتیبانی ارسال کنیم؟

ارسال کنید:

- Diagnostics Redacted.
- وضعیت کلی: Proxy یا VPN.
- اینکه مشکل روی Wi-Fi است یا Mobile Data.
- اینکه Route Check موفق است یا نه.
- تعداد Resolverهای معتبر.
- نسخه برنامه.

ارسال نکنید:

- کلید رمزنگاری.
- رمز SOCKS.
- لینک پروفایل واقعی.
- QR پروفایل.
- لاگ خام شامل مسیرها یا Secrets.
